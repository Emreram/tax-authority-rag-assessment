# Explicit Assumptions — Tax Authority RAG System

> These assumptions are stated upfront in the submission to demonstrate professional maturity
> and to scope design decisions. Each assumption influences specific architectural choices.

---

## Deployment & Infrastructure

| # | Assumption | Architectural Impact |
|---|-----------|---------------------|
| A1 | **Deployment target is government cloud** (Azure Government NL, AWS GovCloud, or on-premises data center). No public cloud SaaS is acceptable for classified data. | All components must be self-hostable. Eliminates Pinecone, Weaviate Cloud, Cohere API as options. Drives OpenSearch + self-hosted models decision. |
| A2 | **No data may leave national jurisdiction** (EU/NL data sovereignty, GDPR Article 44+). Tax authority data including FIOD fraud investigation material is classified. | Embedding models, rerankers, and LLMs must run on-premises or in a government-approved cloud region. No API calls to US-based services. |
| A3 | **GPU infrastructure is available** for self-hosted embedding, reranking, and LLM inference. Minimum: 4× NVIDIA A100 80GB (or equivalent) for LLM, 2× A10G for embedding + reranker. | Enables self-hosted Mixtral 8x22B or Llama 3.1 70B. Without GPUs, fall back to Azure OpenAI Government Cloud. |
| A4 | **The organization has an existing Identity Provider** (Active Directory / ADFS / Azure AD with OIDC). Users are already assigned to organizational groups. | RBAC maps IdP groups → OpenSearch DLS roles. No new authentication system needed. JWT/OIDC integration at the API gateway. |

## Corpus & Language

| # | Assumption | Architectural Impact |
|---|-----------|---------------------|
| A5 | **Primary corpus language is Dutch**, with some English EU regulations and CJEU case law. | Use `multilingual-e5-large` for embeddings (not English-only models). BM25 analyzer uses Dutch stemming and stop words. |
| A6 | **500,000 documents average ~40 chunks each**, yielding ~20 million total chunks. | Drives HNSW parameterization (m=16), quantization strategy (fp16 primary ~61 GB total / SQ8 fallback ~31 GB), shard count (6), and memory planning. |
| A7 | **Documents are ingested in batch** (nightly or on-change re-index), not real-time streaming. New legislation, updated policies, and new rulings are published on a known schedule. | Batch IngestionPipeline design. Deterministic chunk IDs enable upsert on re-index. No streaming architecture needed. |
| A8 | **Legal documents follow the standard Dutch legislative structure**: Wet (Act) → Hoofdstuk (Chapter) → Afdeling (Section) → Artikel (Article) → Lid (Paragraph) → Sub (Sub-paragraph). | Structure-aware chunking parser uses these boundaries. Regex patterns target Dutch legal formatting conventions. |
| A9 | **Case law uses ECLI identifiers** (European Case Law Identifier) as the standard reference system. Format: `ECLI:NL:{court}:{year}:{number}`. | Exact-ID retrieval shortcut: queries containing ECLI patterns bypass vector search and use direct keyword filtering. |

## Users & Scale

| # | Assumption | Architectural Impact |
|---|-----------|---------------------|
| A10 | **System serves hundreds of concurrent users**, not thousands. Peak load: ~200-500 simultaneous queries. | Single OpenSearch cluster (3-5 nodes) is sufficient. No extreme horizontal scaling needed. Latency budget is tight but achievable. |
| A11 | **Three primary user personas**: Tax Inspectors (complex legal queries, need RESTRICTED access), Legal Counsel (case law heavy, need RESTRICTED access), Helpdesk Staff (FAQ-heavy, PUBLIC + INTERNAL only). | Helpdesk benefits most from semantic cache (FAQ patterns). Inspectors need complex query decomposition. Legal counsel needs ECLI exact-match. |
| A12 | **Users expect exact citations** in the format: "Article X, Paragraph Y of [Law Name]" or "ECLI:NL:HR:2023:1234, consideration 3.2". Vague references are unacceptable. | Citation is reconstructed from chunk metadata (hierarchy_path, article_num, paragraph_num), NOT generated by the LLM. Post-generation validation checks cited chunk_ids against retrieved context. |

## System Requirements

| # | Assumption | Architectural Impact |
|---|-----------|---------------------|
| A13 | **TTFT (Time to First Token) < 1.5 seconds** is a hard requirement measured at p95, not p50. | Aggressive latency budgeting. Cache check: 15ms, Embedding: 30ms, Retrieval: 150ms, Rerank: 200ms, LLM first token: 800ms, Buffer: 305ms. |
| A14 | **"Zero-hallucination tolerance"** is interpreted as: the system must never present fabricated legal citations or invent legal provisions. It MAY refuse to answer rather than risk inaccuracy. | Fail-closed CRAG design: Irrelevant retrieval → refuse. Ambiguous → rewrite once then refuse. Generated citations are validated against retrieved context. Unverifiable citations → refuse. |
| A15 | **The assessment expects conceptual architecture + pseudo-code**, not a production-deployable codebase. Pseudo-code should be clean, framework-aware (LlamaIndex, LangGraph), and near-runnable. | Focus on clarity and correctness of design, not production error handling or edge cases. Pseudo-code uses real class names and API patterns from LlamaIndex/LangGraph. |

## Design Philosophy

| # | Assumption | Architectural Impact |
|---|-----------|---------------------|
| A16 | **Prefer false negatives over false positives.** It is better to say "I don't have enough information to answer this with certainty" than to give a potentially wrong fiscal answer. | CRAG grader has a high bar for "RELEVANT". Cache threshold is conservative (0.97). Generation refuses when citations can't be verified. |
| A17 | **Security is a first-class architectural concern**, not a bolt-on. The FIOD classification example in the assessment signals that the evaluator will specifically check for robust access control. | Pre-retrieval DLS enforcement is non-negotiable. Cache is role-partitioned. Security is discussed in every module, not just Module 4. |
| A18 | **The system will be audited.** Government systems require audit trails. Every query, retrieval, generation, and access decision should be logged. | OpenTelemetry traces for every pipeline execution. OpenSearch audit log for access events. Structured JSON logging for all components. |

## How to use these assumptions in the submission

1. **State them upfront** in an "Assumptions" section before Module 1
2. **Reference them** when making design decisions: "Given Assumption A2 (data sovereignty), we select self-hosted OpenSearch rather than a SaaS vector database"
3. **Acknowledge uncertainty**: "If Assumption A3 (GPU availability) does not hold, the fallback is Azure OpenAI Government Cloud (see Section 2.1)"
4. **Show flexibility**: For each critical assumption, briefly mention what changes if the assumption is wrong
